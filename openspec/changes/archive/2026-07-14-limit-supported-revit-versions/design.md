## Context

The repository policy names Revit 2025-2027, but `Relay.sln` and `Relay.csproj` still contain R21-R24 configurations, .NET Framework targets, old Dynamo package versions, legacy conditional symbols, and excluded `Relay.Revit20xx` folders. Post-build output follows whichever configuration is selected, so unsupported artifacts can still be produced.

## Goals / Non-Goals

**Goals:**

- Make the declared and buildable matrix exactly Revit 2025-2027.
- Keep target framework, Revit API, and DynamoRevit package selection explicit per supported host.
- Prevent unsupported release folders from being generated by project configurations.

**Non-Goals:**

- Make one binary run across Revit versions or migrate old installations.
- Guarantee user graph/package compatibility between Dynamo releases.

## Decisions

1. Delete R21-R24 solution/project configurations rather than hiding them from documentation. A support policy that still compiles unsupported binaries remains ambiguous.
2. Remove legacy source directories and conditional branches only after `rg` confirms no R25-R27 code path references them. Keeping excluded dead folders creates false ownership and maintenance cost.
3. Retain independent R25, R26, and R27 properties because their target frameworks and DynamoRevit package versions differ. A generalized formula would conceal compatibility decisions.
4. Add CI matrix builds for Debug R25-R27 and verify Release R25-R27 locally or in release automation where package availability permits. Build outputs must map only to `_Release/Revit2025`, `Revit2026`, and `Revit2027`.
5. Document the last release supporting Revit 2021-2024 rather than shipping compatibility shims.

## Risks / Trade-offs

- [Downstream forks rely on removed configurations] -> Mark the change breaking and retain history/tags for old hosts.
- [R27 package metadata changes] -> Keep R27 values isolated and update them through a dedicated compatibility change.
- [CI cannot restore proprietary/pre-release packages] -> Fail explicitly or document the temporary external prerequisite; do not silently skip a supported target.

## Migration Plan

Remove configurations and legacy code in one change, build all supported configurations, and inspect release paths before publishing. Rollback is a git revert; installed older binaries remain unaffected.
